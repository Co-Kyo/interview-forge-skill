# InterviewForge 目标逆向拆解指南

> 本文件供 WorkBuddy Agent 在 Step 2 出题时读取，指导从测试目标逆向拆解到知识组件，再到题目设计。
> 遵循 Roadmap + 分布式 Prompt 架构，独立于 SKILL.md 主流程维护。

---

## 核心逻辑

```
测试目标（用户要验证什么）
    │
    ▼ 逆向拆解
知识组件清单（要覆盖哪些最小知识单元）
    │
    ▼ 证据映射
每道题需要收集什么证据（证明用户会/不会）
    │
    ▼ 题目设计
生成具体题目（选择题/开放题）
```

这不是"想一道出一道"，而是**从目标出发，有依据地设计每一道题**。

---

## 第一步：从测试目标逆向拆解知识组件

### 什么是知识组件？

知识组件是**最小的可独立验证的知识单元**。一个知识组件满足：
- 可以用一道题独立验证用户是否掌握
- 不能再拆成更小的有意义的测试单元

### 拆解方法：目标 → 问题链 → 组件

从测试目标出发，问自己：**"要验证这个目标，用户需要掌握哪些东西？"** 逐层拆解，直到每个子项都可以用一道题验证。

**示例**：

| 测试目标 | 一层拆解 | 二层拆解（知识组件） |
|---------|---------|------------------|
| 面试叙事：讲清 Vue3 响应式机制 | 1. 依赖收集机制 | 1a. effect 注册时机 1b. track 的触发条件 1c. targetMap 数据结构 1d. dep 集合的增删规则 |
| | 2. 触发更新机制 | 2a. trigger 的触发条件 2b. scheduler 的去重策略 2c. 批量更新的微任务调度 |
| | 3. 核心 API 行为 | 3a. ref 的 getter/setter 3b. reactive 的 Proxy 代理 3c. computed 的惰性求值 3d. watch vs watchEffect |

### 拆解规则

1. **覆盖完整性**：拆出的组件集合必须能覆盖测试目标的全部范围。验证方法：如果用户掌握了所有组件，是否就能达成测试目标？如果是，拆解完整。
2. **最小化原则**：组件数量控制在 8~15 个（对应 8~12 道题的题量范围）。太多说明拆得太细，太少说明拆得太粗。
3. **独立可验证**：每个组件可以用一道题独立测试，不依赖其他组件的答题结果。
4. **测试目标对齐**：不同测试目标的拆解重点不同——
   - **面试叙事**：按因果链拆解（A→B→C，每个环节一个组件）
   - **诊断定位**：按知识域拆解（覆盖面优先，每个域一个组件）
   - **概念辨析**：按差异维度拆解（每个维度一个组件）
   - **案例推导**：按推理步骤拆解（每个步骤一个组件）
   - **知识校准**：按掌握层级拆解（表层/中层/深层各一个组件）

## 第二步：从知识组件映射到证据需求

每个知识组件需要明确：**用户做到什么才算"掌握了"？** 这就是证据需求。

### 证据类型

| 证据类型 | 含义 | 适用题型 |
|---------|------|---------|
| **识别** | 能从多个选项中认出正确答案 | 选择题 |
| **辨析** | 能区分相似但不同的概念 | 选择题（极高相似度干扰项） |
| **解释** | 能用自己的话说明原理或机制 | 开放题 |
| **应用** | 能在给定场景中正确运用原理 | 选择题（场景化）或开放题 |
| **推导** | 能从已知条件推导出结论 | 开放题 |

### 测试目标 → 证据类型偏好

| 测试目标 | 核心证据类型 | 辅助证据类型 |
|---------|------------|------------|
| 面试叙事 | 解释 + 推导 | 识别 + 辨析 |
| 诊断定位 | 识别 | 辨析 + 解释 |
| 概念辨析 | 辨析 | 识别 |
| 案例推导 | 应用 + 推导 | 解释 |
| 知识校准 | 识别 + 解释 | 应用 |

### 映射示例

- 知识组件：effect 注册时机 → 证据需求：解释"effect(fn) 调用后发生了什么" → 题型：开放题
- 知识组件：track 的触发条件 → 证据需求：识别哪种操作触发 track → 题型：选择题

## 第三步：从证据需求设计题目

### 设计规则

1. **一题一组件**：每道题只测一个知识组件（选择题尤其如此）。开放题可以覆盖 2~3 个关联组件，但必须有一个核心组件。
2. **证据类型决定题型**：识别/辨析 → 选择题，解释/推导 → 开放题，应用 → 看场景复杂度决定。
3. **干扰项从"常见误解"中取**：
   - 面试叙事型：干扰项来自"半懂"的理解（看似正确但关键点出错）
   - 概念辨析型：干扰项是相似概念的属性（只差一个限定词）
   - 知识校准型：干扰项是"看似高级但实际错误"的表述（陷阱项）
   - 诊断定位型：干扰项覆盖不同知识域的错误（扫描面广）
   - 案例推导型：干扰项是错误推理链的终点（推理方向错误）

### 选项设计模板

**面试叙事型选项**（考察理解深度）：
```
A. [表面上正确但忽略了关键机制的表述] ← 半懂陷阱
B. [正确答案，精确描述机制]           ← 正确
C. [概念正确但因果倒置的表述]         ← 推理陷阱
D. [混淆了相似系统的表述]             ← 术语陷阱
```

**概念辨析型选项**（极高相似度）：
```
A. [概念X的属性a + 限定词p1]
B. [概念X的属性a + 限定词p2] ← 只改了限定词
C. [概念Y的属性a + 限定词p1] ← 只改了主语
D. [概念Y的属性a + 限定词p2] ← 两者都改
```
正确答案是其中一个，其余三个都是"看起来也对"的高相似度干扰。

**知识校准型选项**（达克效应探测）：
```
A. [高级术语包装的错误表述]    ← 虚高自信陷阱
B. [朴素但正确的表述]          ← 正确（选这个说明真懂）
C. [部分正确的半截表述]        ← 半懂陷阱
D. [明显错误但"看起来很专业"]  ← 蒙对陷阱
```
关键：如果用户选了 A，说明他"以为懂但实际不懂"——用了高级术语但内容错了。

### 开放题设计模板

**面试叙事型开放题**：
```
"请用自己的话解释 [机制X] 的工作流程，从 [起点] 到 [终点] 的完整链路。"
```
- 必须要求"用自己的话"，防止背诵式回答
- 指定起止点，给出框架但留出表达空间
- 可追问"如果没有 [关键组件]，会怎样？"

**案例推导型开放题**：
```
"假设 [具体场景]，根据 [原理X]，请推导 [会发生什么]，并解释你的推理过程。"
```
- 场景必须具体（有参数、有条件）
- 要求解释推理过程，不只看结论

## 第四步：质量自检

题目设计完成后，分两步检查：

### 拆解维度检查（本文件特有）

- [ ] 每个知识组件是否至少有 1 道题覆盖？是否有组件遗漏？
- [ ] 每道题的证据类型是否与知识组件的需求匹配？
- [ ] 是否有题目测了未列入组件的知识？（如有，判断是否补充组件或删除题目）

### 通用质量检查

按 SKILL.md Step 2 的质量检查清单逐项执行。
